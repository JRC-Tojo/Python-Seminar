## TypeError
TypeErrorというエラーは「型（文字列，数字，リストや辞書）の種類が違いませんか？」というエラー

**チェック項目**
- 誤った型の引数を渡していないか
- 引数の数が正しいか
- タプルの要素が一個の時，末尾にカンマがあるか（末尾にカンマがないと，数値扱いになる）

- `list indices must be integers or slices, not str`		
  - リストに [] するときは文字列は使えない．数字かスライスなどに変更してください．

- `int() argument must be a string, a bytes-like object or a number, not 'list'`
  - int() 関数に使うことはできないものを渡した時に表示されるエラー。
  - int()関数は、文字列、bytes-like オブジェクト、数値を渡すことができる。リストの要素をそれぞれ数値にしたい場合に、今回のようにリストをそのままint()に渡すことはできない。

- `'list' object is not callable` 　
  - 呼び出し可能（callable）ではないオブジェクトに対し、呼び出しをすると発生するエラー
  - 呼び出し可能オブジェクトとは、関数やメソッドなどのように() を付けて呼び出すことができるオブジェクト。

- `can only concatenate tuple (not "int") to tuple` 
  - タプルとタプル以外（int）を結合しようとしていることが原因で発生するエラー

- `must be str, not int`
  - 文字列と数値を結合する際に発生するエラー
 
- `'int' object is not iterable`
  - 繰り返しに使用できないオブジェクトを繰り返し処理で使用してしまっている際に発生するエラー　


## IndexError

リストで宣言されていないインデックスへアクセスしようとしたときに発生

**チェック項目**
- リストの長さ（サイズ）
- アクセスしようとしているインデックス
- サイズ外のインデックスにアクセスしようとしていないか
- リストの追加や削除、がどこかで抜け落ちていないか


## KeyError

存在しない辞書のキーを参照しているエラー

文字を参照していてエラーになった場合,　インデックス指定にすると解決するかも


## ValueError

演算子や関数が、正しい型だが適切でない値を持つ引数を受け取ったときに発生するエラー


## SyntaxError

SyntaxErrorはプログラムがPythonの文法として正しくないときに発生するエラー

**チェック項目**
- Pythonのコマンドや関数の名前を打ち間違えていないか
- カッコを閉じ忘れていないか
- 全角スペースを使っていないか（# でコメントを追加する際も,　全角スペースを使うとエラーになる）
- for文, if文, 関数などの「:」を書き忘れていないか
- ''（シングルクォーテーション）や""（ダブルクォーテーション）を忘れていないか

- `invalid syntax`　　
  - 無効な構文である際に発生するエラー
- `EOL while scanning string literal`　　
  - 文字列の定義に文法ミスがあるときに発生するエラー
- `unexpected EOF while parsing`　　
  - 関数やメソッドの括弧を閉じ忘れている際に発生するエラー


## IndentationError

IndentationError は、正しくないインデントが書かれた場合に発生する構文エラー

ex) インデントの空白の数が揃ってない（Tabを使うと発生しにくい）


## NameError

未定義の変数を参照しているエラー

**チェック項目**
- 変数を定義しているか
- スペルミスをしていないか（大文字、小文字を間違えていないか）
- 文字列がシングルクォーテーション（')かダブルクォーテーション（"）のどちらかで囲まれているか
- クラスのメソッドにおいて第一引数にselfを書き忘れていないか
- インポートしていないモジュールを使っていないか


## エラーが起きても実行が止まらないようにする

エラーが起きそうなプログラムに対して、前もってエラーの原因をつぶしておくことはもちろん大切

ただ、想定外のエラーが起きる可能性を排除できないこともあり、そのような場合であっても実行を止めないようにする

- `try ~ except文`

  例えば次のような使い方ができる
  ```py
   list1 = [1, 2, 3]
   try:
   	# 通常はlist1に4番目の値は登録されていないため、IndexErrorが発生して処理が止まる
   	forthVal = list1[4]
   except:
   	# この構文を使うとtryで書かれた処理にエラーが起きた場合、exceptの中に書かれたプログラムが実行される
   	print('4番目の値はlist1に存在しなかったよ')
  ```

  もちろん、これは例なので、エラーが起きても適当なメッセージをprintしているだけだが、本来は、この`except`の中にIndexErrorが起きた時の適切な処理を記述する
  - `except`に中身を何も書かないこと（`pass`とのみ記述するなど）は避けるべき
  - `try ~ except`は処理負荷が大きく，むやみに使うとパフォーマンス低下の原因になるだけでなく，[* 問題があったら別の処理をする]という`try ~ except`の設計思想からも外れるものとなる
  - そのような使い方をするのであれば，事前にif文などでエラーの原因をはじいてから`try`の中に書いていたプログラムを実行すべき（つまり，`try ~ except`を使わない）
  - `try ~ except`を使うときは、できるだけtryの範囲を短くし、プログラム全体をtryに入れるようなことはしない
    - 基本的にエラーは起きてはいけないもののため、起きることが異常であることに変わりはない
    - この構文はあくまで、この前提を理解したうえで様々な処理に対応するための一助として使用するように心がける


## 自分でエラーを定義する

プログラムを書いていると、処理の途中でこれ以上先に進まず、実行を中断してほしい時がある

例えば以下のような場合
- 計算のルール的に（＝制約条件的に）特定の値が入ってきたら処理を止めてほしい場合
- 関数の引数に開発時には想定していない値が入ってきた場合
- などなど。。。

このようなときには`raise 適切なエラー`として自分でエラーを出す必要がある

`raise1.py`
```py
list1 = [1, 2, 3]
# 何かしらの影響で-1がリストに入ったとする
list1[0] = -1

# だが、記述しているプログラムはリストに負の数が入ってほしくない時、値に問題がある旨を示すエラーを出す
if any((x < 0 for x in list1)):
  raise ValueError('list1には負の数が入ってはいけません')
```

`raise2.py`
```py
def func1(arg1:list):
  # arg1がlistでない場合にエラーを出す
  if not isinstance(arg1, list):
    raise TypeError('arg1はlistでなければいけません')
```

- このようにエラーを出すことによって、不本意な値で後ろの処理が実行されることがなくなる
- `raise`の後ろに書いているエラーはこのページの前半にあるようなPythonに標準で定義されているエラーを用いる（どうしても標準のエラーでは表現できない場合、[自作する](https://helve-blog.com/posts/python/python-user-defined-exceptions/)こともできる）
- 基本的にはエラーの理由に沿ったエラーを選択することが理解のしやすさという面で大切


## assertについて

`assert`は特定の条件に合致しなかった場合にエラーを出して処理を止めることができるというもの
```py
# raise1.pyをassertを使って書いてみる
list1 = [1, 2, 3]
list1[0] = -1

assert all((x > 0 for x in list1)), 'list1には負の数が入ってはいけません'
```

このようにすることで、list1に負の数が入った場合、後ろの処理が実行されなくなる

`assert`の条件式には「本来はこうであるべき」な内容（＝今回は全ての値が正）を条件式として記述する

ifを書く必要がなくなるため、コードの記述量的には`assert`を安易に使いたくなるが、**使いどころをよく考えて**使用する
- assertを使って書いたエラーは全てエラー理由が`AssertionError`となってしまい、エラー原因を探る理由がメッセージだけになってしまう
- この記事の前半でも書いてある通り、○○Errorの部分はそのエラーがざっくりどんな内容か知る上でとても大切
- raiseの例で記述した、変数の型チェックのようなユーザーの利用方法によって起きる可能性のあるエラー（＝予見可能なエラー）は`raise TypeError`のようにエラーの内容に即したエラーを使用するべき
- そのため、`assert`は[* 絶対に起きないはずの条件だけど、万が一起きたらそれはユーザーの入力やほかのプログラムの書き方とは一切関係のないプログラムのバグ]というときにしか使わない
  - 頻繁に`AssertionError`が起きる時点でそれは`raise`を使った書き方にしたほうが後々エラーの内容が分かりやすい
- つまり、我々のようなデータ解析を主とするプログラムでは基本的に`raise`を使用し、プログラムそのもののデバッグの際にのみ`assert`を使用する
